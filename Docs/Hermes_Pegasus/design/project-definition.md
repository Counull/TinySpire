# TinySpire · 项目与游戏定义

> 本文档回答一个最基础的问题：**我们现在到底在做什么项目、什么游戏、什么范围。**
>
> 如果后续文档、Agent 或实现计划与本文档冲突，先以本文档为当前项目定义，再决定是否显式修改本文档。

## 1. 项目定义

| 项 | 当前定义 |
|---|---|
| 项目代号 | TinySpire |
| 引擎 | Unity 6.5 |
| 当前形态 | Demo / 垂直切片 / 学习型独立项目 |
| 游戏类型 | Slay the Spire-like roguelike deckbuilder，联机 PVE 方向 |
| 当前阶段 | BattleScene MVP planning / early implementation |
| 当前主目标 | 跑通 **Card → Effect → BattleState → UI → Feedback** 的最小闭环 |

TinySpire 不是先做完整商业化内容量的独立游戏，而是先做一个足够小的战斗垂直切片：一张牌能被打出，一个效果能进入状态，一个状态变化能驱动 UI 和反馈。

## 2. 游戏一句话定义

> **TinySpire 是一个以多人协作为核心增量的杀戮尖塔式卡牌肉鸽：每个玩家拥有独立英雄与牌组，在共享敌人阵列上通过交错出牌和跨玩家状态引用形成 combo。**

这句话里有三个不可省略的核心：

1. **杀戮尖塔式卡牌肉鸽**：抽牌、弃牌、牌组循环、路线/成长压力、卡牌升级。
2. **多人协作 PVE**：不是 PVP，不靠信息不对称制造张力。
3. **跨玩家 combo**：联机价值不是四个人各打各的，而是战士挂破甲、法师上火、游侠触发等跨玩家状态引用。

## 3. 当前 MVP 定义

当前 MVP 只验证战斗闭环，不验证完整 roguelike 循环。

### 必须包含

- 一个 BattleScene。
- 至少一名玩家的手牌展示。
- 至少一张可打出的测试牌。
- Card → Effect → BattleState 的执行链。
- BattleState → UI 的可观察更新。
- 至少一个明确反馈：数值变化、buff 标记、hit shake、简单 VFX 之一。

### 可以暂不包含

- 完整地图路线。
- 商店。
- 遗物系统。
- 完整职业池。
- 完整联网实现。
- 完整美术资产。
- 完整构筑循环。
- 完整敌人 AI。

## 4. 当前玩法基线

来自 `baseline.md` 与 `decisions.md` 的当前基线：

- 每人独立英雄。
- 每人独立牌组。
- 共享敌人阵列。
- 队友手牌可见。
- 回合内一张牌就切人；不是一个人打完整回合才切。
- 敌人只在所有玩家完成本回合行动后行动。
- 战利品共享池、路线事件投票是参考方向，但不属于 BattleScene MVP。
- 卡牌是 template + instance 两层模型。
- 卡牌升级存在；升级窗口当前选为篝火处，每人独立选择。
- 抽牌堆抽空后自动洗牌。
- 手牌上限固定，暂定 10。
- 每回合抽牌数固定。
- 默认不允许自由送牌/交易；后续可以通过特定卡牌效果引入。

## 5. 非目标 / 不要误解成当前要做

这些东西可以成为未来游戏的一部分，但不是当前阶段要实现的承诺：

- “多人联机 PVE”不等于现在就做完整网络同步。
- “卡牌肉鸽”不等于现在就做地图、事件、商店、遗物。
- “构成主义奇幻”只是 Demo / 第一章视觉方向，不锁死整个世界观。
- “职业不重叠”是跨玩家 combo 的设计原则，不等于现在立刻产出完整职业表。
- “保留接口”不等于现在实现功能。

## 6. 文档优先级

当前设计文档建议按以下顺序读取：

1. `project-definition.md` — 当前项目和游戏定义。
2. `decision-locks.md` — 哪些决定已经锁定、哪些只是暂定、哪些仍开放。
3. `baseline.md` — 原始基础设定和参考对象。
4. `decisions.md` — 具体玩法决策记录。
5. `../architecture.md` — Unity/C# 架构基线。

## 7. 改动规则

如果要改变本文档里的核心定义，不要直接顺手改代码或其他文档。

正确流程：

1. 在 `decision-locks.md` 找到相关锁定项。
2. 如果是 Locked，先写明为什么要 reopen。
3. 在 `decisions.md` 新增一条替代决策或修订记录。
4. 再修改本文档和受影响的实现计划。
