# TinySpire · 决策锁定表

> 本文档不是“永远不能改”的神谕，而是防止项目每天换地基。
>
> **Locked 的意思是：不能在实现中顺手改变；如果要改，必须显式 reopen，并说明改动会影响哪些文档、代码和 MVP 范围。**

## 1. 锁定等级

| 等级 | 含义 | 改动规则 |
|---|---|---|
| Locked | 当前项目地基。实现、计划、Agent 输出都必须默认遵守。 | 只能通过新增/修订决策来 reopen。 |
| Provisional | 当前先按这个做，但数值或细节允许在实现中验证。 | 可以在同主题决策下补充修订记录。 |
| Deferred | 明确暂不做，但保留未来入口。 | 不进入当前 MVP，除非用户重新排优先级。 |
| Open | 还没决定。 | 不应被代码或文档假装成已决定。 |

## 2. 当前 Locked 决策

这些是当前设计地基，不应被普通实现任务改变。

### L-001：游戏类型与协作核心

- 来源：`baseline.md`，`project-definition.md`
- 当前结论：Slay the Spire-like 卡牌肉鸽，多人联机 PVE 方向。
- 锁定点：联机价值来自跨玩家 combo，而不是四个单人牌组并排运行。
- 影响：职业设计、敌人阵列、状态系统、UI 可见性。

### L-002：每人独立英雄 + 独立牌组

- 来源：`baseline.md`
- 当前结论：每个玩家有自己的英雄和牌组。
- 锁定点：默认不做全队共享手牌或共享牌组。
- 影响：卡牌归属、牌组构建、网络同步、UI 分区。

### L-003：共享敌人阵列为基础

- 来源：`decisions.md` 决策 010
- 当前结论：四名玩家面对同一组敌人，状态可以跨玩家引用。
- 锁定点：默认不是每人一套独立敌人。
- 例外：Boss / 机制战可以出现分血条或局部独立目标。

### L-004：队友手牌可见

- 来源：`decisions.md` 决策 006
- 当前结论：联机 PVE 中队友手牌可见。
- 锁定点：不通过隐藏队友手牌制造难度。
- 理由：不可见只会把信息同步压力转移到语音/打字。

### L-005：回合内交错出牌

- 来源：`baseline.md`，`decisions.md` 决策 008
- 当前结论：一张牌就切人 / 谁想好了谁打；不是一人打完整回合才轮到下一个人。
- 锁定点：避免四倍等待时间，保留跨玩家 combo 的自然感。
- 注意：具体输入仲裁和网络同步策略属于实现细节，另行设计。

### L-006：卡牌 template + instance 两层模型

- 来源：`decisions.md` 决策 001
- 当前结论：卡牌模板与卡牌实例分离。
- 锁定点：一张“打击”的两个副本可以拥有不同实例状态。
- 影响：升级、临时修改、网络同步、存档。

### L-007：存在卡牌升级

- 来源：`decisions.md` 决策 001 / 002
- 当前结论：项目默认有卡牌升级；升级使 instance 模型必要。
- 锁定点：不能把卡牌实现成纯模板引用后再假装支持升级。

## 3. 当前 Provisional 决策

这些先按当前结论推进，但允许在 MVP 验证后调整。

### P-001：升级时机为篝火处，每人独立选

- 来源：`decisions.md` 决策 002
- 当前结论：篝火处升级；回血 vs 升级形成路线压力。
- 暂定原因：BattleScene MVP 暂不做完整地图/篝火，因此短期只影响数据模型，不影响首个切片。

### P-002：牌组循环为自动洗牌

- 来源：`decisions.md` 决策 003
- 当前结论：抽牌堆抽空后自动洗弃牌堆。
- 暂定原因：符合标准肉鸽；除非后续发现等待/节奏问题，否则按此做。

### P-003：手牌上限固定，暂定 10

- 来源：`decisions.md` 决策 004
- 当前结论：手牌上限不做职业差异化，数值暂定 10。
- 暂定原因：10 是数值占位，不是最终平衡结果。

### P-004：每回合抽牌数固定

- 来源：`decisions.md` 决策 005
- 当前结论：基础抽牌数固定；卡牌/遗物以后可以改变抽牌。
- 暂定原因：具体抽几张仍属数值阶段。

### P-005：超时处理为跳过回合

- 来源：`decisions.md` 决策 009
- 当前结论：有人卡住时跳过回合，有过期时间提示，不做 AI 自动操作。
- 暂定原因：需要联网体验验证；当前 BattleScene MVP 可以先不实现。

## 4. 当前 Deferred 决策

这些明确暂不进入当前 MVP。

### D-001：默认送牌 / 交易

- 来源：`decisions.md` 决策 007
- 当前结论：暂不做默认跨玩家传牌，保留接口。
- 未来入口：特定卡牌效果可以允许“将一张手牌交给队友”。
- 当前规则：不要把自由交易做进基础操作。

### D-002：完整联网实现

- 来源：`project-definition.md` / `AGENT_HANDOFF.md`
- 当前结论：TinySpire 是联机 PVE 方向，但当前 MVP 不优先做完整网络同步。
- 当前规则：可以按联机友好的数据边界设计，但不要把 MVP 扩成联网工程。

### D-003：完整 roguelike 外层循环

- 来源：`project-definition.md` / `AGENT_HANDOFF.md`
- 当前结论：地图、商店、事件、遗物、完整构筑循环都暂不优先。
- 当前规则：BattleScene 先跑通状态链路。

## 5. 当前 Open Questions

这些还没定，不能被实现任务假装成已决定。

- 几人队伍是固定 4 人，还是 1–4 人可变？
- 职业划分有哪些？
- 是否允许重复职业？如果不允许，是设计限制还是匹配限制？
- BattleScene 第一版 UI 使用 UI Toolkit 还是 UGUI？
- 卡牌数据第一版用 ScriptableObject、JSON，还是纯 C# mock？
- 是否从第一天引入 VContainer / R3 / UniTask / DOTween？
- 敌人意图、行动序列、仇恨/目标选择模型如何定义？
- 跨玩家状态引用的最小规则是什么？例如破甲、燃烧、标记的归属和触发权。
